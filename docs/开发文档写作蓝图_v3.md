## 零、一个决定全局的决策：设计全景 / 实现核心 分离

这是让文档好写、又不被答辩问穿的关键一招。

- **文档层面**：完整设计全部 7 个模块 —— 用宽度撑起 150 页，每个模块都有需求、设计、E-R、状态图。
- **实现层面**：只真实做通**核心闭环**，并在"详细设计-实现结果"里**诚实标注**哪些是本期实现、哪些是设计但简化/未实现。

| 范围 | 模块 | 本期状态 |
|---|---|---|
| **本期实现（演示跑通）** | M1 认证（学号简易核验+角色权限）、M3 知识库（CRUD+搜索+生命周期状态机）、M4 结构化求助（求助单+模板回答+**路由匹配**+采纳+生成候选）、M7 审核（知识候选+认证，含 checklist）、M6 时间线浏览+个人进度 | 真实代码 |
| **设计完整、实现简化** | M5 机会与组队（设计完整状态机，实现列表+组队）、M2 校友路径推荐（设计规则引擎，实现最简版或 mock 数据） | 部分/mock |
| **设计但本期不做** | 内推私密流程（标注 Could）、真实教务/校友数据源对接 | 仅设计 |

> 答辩话术：「我们的实现聚焦最能体现工程价值的知识沉淀闭环，其余模块完成了完整设计，受课程周期限制作简化实现——这是我们主动的范围管理决策。」这比假装全做完、被现场问穿要高分得多。

## 一、收敛后的项目定义（v3，可直接写进引言）

> 本系统是面向新疆大学在校生与毕业生的**成长导航型信息系统**，核心机制是**结构化知识沉淀闭环**：学生的一次性求助经验，经"求助 → 按标签匹配应答 → 采纳 → 知识候选 → 审核 → 入库 → 被成长时间线与搜索引用"转化为带**适用范围、时效与来源**的可复用知识资产，并由成长时间线按"专业 × 年级 × 发展路线"主动编排推送。系统聚焦**竞赛、大创、考研、就业、组队等高决策成本、强信息差场景**；生活类信息作为只读引流入口、高时效内容一律外链官方渠道。区别于论坛的信息流模式，实现"看路径、查经验、问问题、找机会、跟时间线"的目标导向式服务。

关键收敛动作（对比 v2）：
1. 核心场景**收窄**到高决策成本信息，生活圈降级为只读引流入口。
2. 求助闭环**补上路由机制**（否则闭环空转，是全项目最致命的漏洞）。
3. 认证改为**分级方案**（学号核验 / 学院邀请码），不再是空洞的"双身份认证"四个字。
4. 需求分**真/弱/拿不到**三层，诚实声明不覆盖需匿名的敏感信息。

## 三、逐章写作指南（对着你们的目录）

### 一、引言
- **1.1 开发目的 / 1.2 现状及意义 / 1.3 背景**：写"需求三层"结论——高决策成本场景是真需求（微信群/公众号/小红书做不了结构化沉淀与横向对比），生活信息是弱需求（只做引流），敏感内幕拿不到（诚实声明）。背景里点明信息分散、易被刷走、难沉淀，并引用你们做的需求调研（见第四章）。

### 二、系统可行性分析（这是你们目录里的重头戏，评审素材最多）
- **2.3 条件、假定和限制**：写清关键假定——课程演示用模拟数据、定位为学院试点、认证走学院邀请码而非对接教务。
- **3 对现有系统的分析**：这里放**竞品对比表**（评审的杀手素材）。把 3.1-3.6（处理流程/工作负荷/费用/人员/设备/局限性）重构为"现有替代方案的能力与局限"：微信群、QQ 群、公众号、超级课程表、知乎/B站/小红书、学校官网、校友会，按**免费/可信/迁移成本/可结构化对比/可沉淀检索**五维度逐一列，最后收敛出"没有一个能同时满足结构化沉淀 + 横向对比"的缝隙。
- **4.2 处理流程和数据流程**：放 **DFD 数据流图**（0 层顶层图 + 1/2 层展开"求助→候选→审核→入库"）。
- **4.3 改进之处**：差异化定位——结构化、可横向对比、带时效评价、可被时间线与搜索引用的知识资产。
- **4.6 技术条件方面的可行性**：单体三层架构（如 Spring Boot + Vue + MySQL）、成熟技术栈、无高风险技术；主动写明"刻意不用微服务/分布式"是范围克制。
- **5 可选择的其他系统方案**：方案 1 = 独立 App 直接 To C；方案 2 = 寄生小程序 / 给学生组织用的迎新内容管理工具（To 学生组织）。对比后说明为何选当前方案（评审直接给了这两个备选）。
- **6 投资及效益分析**：支出（学生云服务器数百元/年、域名）；收益用"信息差造成的时间成本节省"做社会效益估算（如 N 名新生 × 每人节省 M 小时）；回收周期与敏感性分析按社会效益口径写。
- **7.1 法律方面的可行性**：**写 PIPL 合规**（评审盲区 1）——校友去向是可识别个人信息（需单独同意）、证件是敏感个人信息（需加密+最小必要）；本系统用字段级可见性授权、聚合脱敏、演示用模拟数据来满足合规。**这一节是差异化加分点。**
- **7.2 使用方面的可行性**：认证可行性、冷启动方案（种子导入+学院背书）、内容维护责任人机制。
- **8 结论**：可行，聚焦核心闭环，其余简化实现。

### 三、系统开发计划
- **3.1 工作任务分解与人员分工**：5 人 × 业务域 + 横向职责表。
- **3.3 进度**：放**增量迭代计划**（评审致命项的解法）——迭代一打通核心闭环（M1+M3+M4），可独立演示；迭代二三补 M2/M5/M6/M7；每迭代一次集成测试。**画一张里程碑甘特图**（含开题/中期/答辩节点、需求基线冻结点）。
- **2.4 验收标准**：每模块写**可测量**准则（如"种子知识条目 ≥80 条""闭环在演示脚本中完整触发一次""机会状态机四态全部可复现"）——评审专门点名要这个。
- **3.5 关键问题**：内容冷启动、认证强度、并发编辑——写风险 + 对策。

### 四、系统需求分析
- **需求获取（写进 2 任务概述 或单列附录）**：**这是必补的**——补一轮 8-15 人访谈或 50+ 样本问卷（覆盖新生/高年级/毕业生），产出"需求获取纪要"，需求逐条标注来源（"访谈#3""问卷Q7"）。没有这个，"信息差"就是团队臆想，评审必攻。
- **2.1 目标**：写 2-3 条**可测量子目标**（如"FAQ 平均查找时间从 X 降到 Y""候选到发布平均审核时长 ≤Z 小时"）。
- **3.1 系统流程图**：全局系统流程图。
- **3.2 数据流图及用例模型**：**用例图**（三角色：在校生/毕业生/管理员，总图 + 每模块一张）。
- **3.3 数据字典**：核心实体字段定义。
- **需求追溯矩阵**：目标→子目标→功能→测试用例，贯穿全文的主线。

### 五、系统概要设计
- **2.4 软件结构设计**：**软件结构图**（7 模块 + 双圈视图层）。
- **2.5 功能需求与程序的关系**：引用追溯矩阵。
- **5.1 概念结构设计**：**E-R 图**（约 12-15 实体：用户/档案/路径卡/知识条目/求助单/回答/机会/队伍/时间线节点/关联表/**去向版本表**）。关联表体现"只存 ID"的解耦设计；路径卡加**去向类型分支字段**（深造 vs 就业）；加**字段可见性**与**修订留痕**（评审盲区 5 的解法）。
- **5.2 逻辑结构设计**：表结构（含主外键、可见性字段、样本量门槛逻辑）。
- **6 系统出错处理设计**：认证失败、**并发编辑冲突（乐观锁 version 字段）**、去向统计样本不足降级显示。

### 六、系统详细设计（亮点集中地）
- **2 程序系统的结构**：类图/包图。
- **3 设计说明**：重点写两个有算法含量的部分——(a) **求助-校友路由匹配**（按问题类型/目标方向匹配标签校友并通知，给伪代码）；(b) **知识条目生命周期状态机**实现。
- **状态图 × 4**：求助单、知识条目、机会、队伍——**你们的王牌，一定要画全**。
- **时序图**：知识沉淀闭环（求助发布→路由→回答→采纳→候选→审核→入库→被时间线引用），跨 4 模块 3 角色，展示低耦合。
- **活动图（带泳道）**：新生落地流程、审核工作流。
- **4 界面设计**：首页**仪表盘**（我的时间线进度/即将截止机会/本专业高频问题，无信息流）——这是"反贴吧"的直接证据。
- **5 实现结果**：每人负责模块的实现 + 核心代码。

### 七、系统测试
- **3 测试结果**：放**可测量验收准则对应的用例** + 并发编辑冲突用例 + 越权（水平/垂直）测试 + 边界值/异常输入用例。保留真实"失败→修复"记录。
- **5.2 缺陷和限制**：诚实写本期未实现/简化部分——这是加分不是扣分。

### 八、系统用户操作手册
- 三角色（在校生/毕业生/管理员）分别的操作说明，配界面截图。

### 九、总结与展望
- 展望里写：路径推荐引擎完整化、真实数据源对接、**系统退役与交接计划**（交给学院/下一届、停服时数据销毁流程——评审盲区 3，极少人写）。

## 五、让文档好写的五个技巧

1. **一条主线串全文**：知识沉淀闭环是脊柱，每章开头都回指它——需求、DFD、状态图、时序图、测试用例全挂在它上面，写起来不散。
2. **先定术语表和模块编号表**：动笔第一天建一份，全员引用。模块名/字段名/术语在图、表、正文里对不上是评审扣分重灾区。
3. **评审发现直接变章节内容**：不用重新想——竞品分析进可行性 3、PIPL 进 7.1、迭代计划进三、追溯矩阵进四、状态图进六。
4. **先画图后写字**：图定了，正文就是图的说明，最省力。
5. **设计全景 / 实现核心分离，诚实标注**：避免被答辩问穿，同时撑满文档宽度。
