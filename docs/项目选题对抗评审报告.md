# 《新疆大学校友圈与双圈成长导航平台》对抗评审报告

> 评审方式：多 agent 对抗式评审。5 个独立视角（需求真实性 / 需求覆盖度 / 软工流程 / 严苛评审老师 / 用户场景走查）并行挑刺，产出 32 条批评；"魔鬼辩护人 + 终审法官"对每条先做最强反驳再裁决；最后由完备性检查扫全员盲区。
> 裁决统计：**26 条裁决 = 23 条部分成立 + 3 条不成立 + 0 条完全成立**；另有 5 条全员盲区、8 条软工流程补充审查。
> 日期：2026-07-04

---

## 一、总判决

**需求是"部分真实但被范围高估"的；设计骨架能覆盖，但有 4 个机制级缺口；软工过程层整体缺失一层（需求工程）。选题可以活，且经过打磨后强于大多数课程选题——前提是完成本报告第六节的修改清单。**

需求的三分法（这是全部评审收敛出的最重要结论）：

| 需求分层 | 判定 | 说明 |
|---|---|---|
| 高决策成本场景（考研/竞赛/大创/就业经验、校友真实路径、组队） | **真需求** | 信息差真实存在，微信群/公众号/小红书做不了结构化对比与沉淀，是全项目该聚焦的核心 |
| 新生生活信息（校园网/快递/食堂等） | **弱需求** | 群里问一句就解决、时效性上平台结构性劣于官方和当届社群；只配做引流入口，不配做核心卖点 |
| 最高痛点的敏感信息（导师避雷、保研内幕） | **结构性拿不到** | 实名无匿名的设计纪律恰好排除了这类信息；报告中要诚实承认不覆盖，不要宣称 |

## 二、被驳倒的批评（= 答辩弹药，直接可用）

法官裁定 **不成立** 的 3 条，反驳词可直接用于答辩：

1. **"双边冷启动死锁必死"→ 不成立。** 反驳词：运营假设已是"寄生冷启动"——种子内容由团队将现有新生群 FAQ 与公众号攻略结构化导入，管理员由学院老师担任（机构背书），课程演示用模拟数据跑通全流程。首批内容从哪来、谁写、如何演示，均有答案。
2. **"讲座塞进机会类型是镀金"→ 不成立。** 反驳词：讲座只是机会状态机的一个枚举值，复用同一套逻辑，边际成本近乎为零；"不错过机会"正是核心动词"找机会"的一部分。
3. **"现场 kill 一个微服务给我看"→ 不成立。** 反驳词：本系统刻意保持单体架构，通篇没有微服务/分布式/AI/大数据关键词——这种克制恰好规避了"关键词堆砌必有假功能"的雷区。答辩时可主动把这一点讲成设计美德。

另有两条被大幅削弱（答辩时可引用反驳部分）：
- "低频产品必死"：课程大作业不考核 DAU；且选课季/竞赛季/大三分流/大四求职构成四年多触点。**但**必须删除任何依赖日活的变现叙事，定位诚实写成"低频工具型成长手册"。
- "学长必然不贡献"：单校实名小社区 + 学院背书下，"被采纳次数可见 + 帮助人数计数"是有意义的声誉信号，不同于匿名大平台。**但**激励机制必须从 Could 提为 Must（见下）。

## 三、成立的机制级缺口（按杀伤力排序）

这些是"模块存在但驱动它的机制是空的"类问题，全部有法官给出的 must_change：

1. **求助单没有路由机制（击中系统灵魂）。** 校友不会天天刷"求助广场"，缺"按标签匹配→定向推送/通知"这一步，采纳环节进不去，整条知识沉淀闭环空转。→ 在 M4–M2 之间设计求助单路由：创建时按问题类型/目标方向匹配对应标签的校友并产生站内通知；验收标准写"发布后 N 小时内触发至少 1 次匹配通知"。
2. **"明确成长方向"没有决策支持能力。** 大三分流是全平台最高价值场景，现有能力止步于"看别人怎么走"（静态浏览+统计）。→ M2 新增"路径推荐"子需求并升为 Must：输入专业/年级/GPA 区间/兴趣标签/目标方向，规则引擎过滤排序即可；M6 节点加"匹配度/推荐理由"字段。
3. **双身份认证四个字下面是空的。** 无权威数据源时管理员只能目测截图，毕业 3-10 年的校友无法核验，而校友可信是路径卡和内推价值的前提。→ 写明认证分级：在校生走学号核验；毕业生走"学院/辅导员邀请码定向邀请（机构侧确权）"或"邮箱/学号 + 人工 + 同专业 2 人担保"；认证风险写入可行性分析。
4. **贡献者激励被排到 Could，本末倒置。** 决定内容供给的机制排最低优先级。→ 至少一种低成本激励升为 Must：贡献者认证标识 + "已帮助 N 位学弟学妹"计数 + 被采纳次数可见 + 学院/团委承认的荣誉证明；每个校友输入入口标注预计耗时（"约 3 分钟"）。
5. **时间线与"新生不迷茫"自相矛盾。** 用时间线必须先选考研/就业/竞赛/考公——这恰是新生答不出的问题。→ 拆一条"未决策路线"默认时间线（大一通用阶段任务），四条分化路线到大二下/大三上再选，作为进阶视图。
6. **路径卡字段模型只装得下就业叙事。** 考研上岸的目标院校/初试/复试细节没有结构化字段，"考研路线"是一等路线却查不到关键信息。→ 路径卡加"去向类型"分支字段：深造展开院校/成绩/复试字段，就业才展开城市/行业/公司/岗位。
7. **审核工作流没有可运转的设计。** 开学季认证+知识候选双队列堆积，无自动初审、无批量操作、无 checklist，会退化成"一键通过"或"不敢审"。→ M7 加自动初审（学号自动核验，边缘案例转人工）、批量通过/退回+标准理由、"三秒可判断"的隐私 checklist（真实姓名/联系方式/可反向定位组合，勾选即退回）；定义"高峰期最大日审核吞吐量"非功能指标。
8. **认证把新生急用信息锁住了。** 开学周认证队列最堵，恰是校园网/快递需求窗口。→ 认证前可访问只读公共内容（生活圈 FAQ），写操作（发求助/填画像）才要求认证。
9. **去向统计小样本会误导。** 单专业个位数样本 + 幸存者偏差，展示百分比是伪权威。→ 设最低样本门槛：某专业路径卡 <20 条时显示"样本不足，仅供参考"并隐藏百分比。
10. **内推装不进公共机会状态机。** 内推需要"投递→推荐→面试进度→结果"的私密个体追踪。→ 要么单独建模"内推申请单"实体，要么在报告中诚实标注本期不做。
11. **时间线是静态模板。** 缺"当前时间 vs 节点建议时间"比对和"已错过 X，现在优先做 Y"的补救提示。→ 加动态比对逻辑，把时间线从展示表升级为导航工具。
12. **UGC 更新责任人未定义。** → 高时效信息（报到时间/选课截止/政策）一律外链官方、绝不自存；低时效经验类才自沉淀；建立"当届学生认领更新 + 过期自动降权"。

## 四、软工流程审查（8 条，直接决定报告能不能过）

1. **【致命】M1–M7 是解决方案冒充需求。** 状态机、关联 ID、零复制都是设计决策，不是需求。→ 文档拆两层：先写不含实现字眼的需求规格（用户目标/用例/优先级/验收标准），再写概要设计；给出需求→设计显式映射表。
2. **【致命】通篇无需求获取活动。** "信息分散、容易被刷走"三句断言没有访谈/问卷/竞品数据支撑，评审必问采样规模。→ 补 8–15 人访谈或 50+ 样本问卷（覆盖新生/高年级/毕业生），产出"需求获取纪要"附录，需求逐条标注来源（"访谈#3""问卷Q7"）。
3. **【致命】隐含大爆炸式集成的瀑布假设。** 5 人各管一域到期末才合，M4 依赖 M3、M7 依赖 M3/M4/M5，最后两三周发现接口不匹配无时间返工。→ 改增量迭代：迭代一先打通核心闭环（M1 认证 + M3 最小知识库 + M4 求助-采纳闭环），可独立演示；迭代二三补 M2/M5/M6/M7，每迭代做集成测试。
4. **【致命】需求可追溯链断裂。** 从"减少信息差"直接跳到 7 个模块，没有可测量子目标层。→ 补追溯矩阵：目标 → 2-3 条可测量子目标（如"FAQ 平均查找时间从 X 降到 Y""候选到发布平均审核时长 ≤Z 小时"）→ 功能点 → 测试用例编号。填不进矩阵的功能要么补干系人来源，要么砍。
5. **【严重】双圈与反贴吧纪律无用户证据。** "生活/学业"二分和"不像贴吧"目标都是团队自定哲学。→ 用卡片分类法（card sorting）验证二分类是否自然收敛；用受访者原话作为设计依据写入需求文档。
6. **【严重】无里程碑。** → 补迭代计划表：各周次的需求基线冻结点、核心闭环可演示点、模块接入点、测试收尾点、验收演示点。
7. **【严重】验收标准不可测量。** "跑通全流程"无成功边界。→ 每模块写可测量验收准则（如"种子知识条目 ≥80 条""闭环在演示脚本中完整触发一次""机会状态机四态全部可复现"）。
8. **【中等】v2 无基线与变更管理。** "v2"承认有 v1 但无变更记录。→ 冻结 v1.0 基线日期，变更走"申请-影响分析-批准-合并"，保留 v1→v2 diff 作为报告附录（这反而是现成的加分素材）。

另（法官裁决补充）：报告必备但提案缺失的软工要素——技术栈选型与总体架构图、按周工时排期、配置管理方案、测试策略与 CI、150 页报告目录大纲。

## 五、全员盲区（5 条，可行性分析章节的加分素材）

5 个评审都没发现、由完备性检查捕获。对课程大作业而言，这些不是"不做的理由"，而是写进报告风险章节的高分素材：

1. **《个人信息保护法》红线。** 公开聚合"某校友在某公司某岗位"属可识别个人信息（需单独同意）；存储学生证/毕业证属敏感个人信息（需加密+更严保护）。→ 报告写明：演示用模拟数据；真实上线需知情同意、可撤回、最小必要机制；路径卡逐字段授权就是合规设计的体现。
2. **机构合法性。** 学校不会向一学期课程项目开放教务/校友官方数据；平台与就业指导中心、校友办职能重叠，有被视为"冒充官方"叫停的风险。→ 可行性分析中定位为"学院试点、与官方渠道互补"，认证走学院邀请码而非声称对接教务。
3. **课程结束后的交接真空。** 团队解散后服务器、维护、用户数据删除义务无人承接，持有真实认证材料的僵尸平台是法律负债。→ 报告写"系统生命周期与退役计划"一节：交接给学院/下一届，或停服时数据销毁流程。写这一节的课程项目极少，是差异化加分点。
4. **认证徽章被武器化的诈骗场景。** 认证不严时，作恶者骗取"认证校友"标识后对新生精准投放内推诈骗/培训贷。→ 威胁模型中加入"认证被反向利用"场景：校友发布机会类内容需二次审核、外链白名单、涉及收费的内容一律拦截。
5. **数据口径与可追溯性。** 去向分类（考研/保研/就业/考公/出国/灵活就业）口径不统一则跨专业跨届不可比；去向随跳槽变化无版本留痕则统计不可审计。→ 定义统一去向口径枚举 + 路径卡修改留痕（修订历史），E-R 图里加版本表——又是一个现成的加分设计。

## 七、评审老师的 5 个刁钻问题（提前准备）

1. 两个用户同时改同一条记录谁赢？→ 准备乐观锁 version 字段方案 + 双账号现场复现。
2. 改 URL 的 id 能不能看别人的隐私路径卡？→ 后端接口级鉴权 + 越权测试截图留证。
3. 测试覆盖率是真在 CI 跑还是事后凑数？→ 保留失败→修复的 commit 历史，现场打开 CI。
4. 去掉知识沉淀闭环系统还剩什么？→ 承认闭环就是灵魂，用完整数据流演示端到端走通（这题对你们是送分题，前提是路由机制补上了）。
5. 新生入学第 60 天为什么再打开你的产品？→ 用"四年多触点"回答：选课季查课程经验、竞赛季看机会截止、大三看去向统计、大四查求职路径；不要用 DAU 叙事硬顶。
