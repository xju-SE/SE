## 二、测试用例清单（14条，TC-01 ~ TC-14）

| 用例编号 | 功能点 | 前置条件 | 测试步骤 | 预期结果 |
|---|---|---|---|---|
| TC-01 | 注册登录 + JWT签发 | 数据库无该用户名记录 | ①`/register`提交合法信息②`/login`用同一账号登录 | 注册成功写入`user`表(密码BCrypt加密非明文)；登录返回`Result{code:200}`且`data`含accessToken/refreshToken；后续请求头`Authorization: Bearer <token>`可访问受保护接口 |
| TC-02 | 发求助 + 路由匹配通知 | 已存在多名不同专业/年级/被采纳历史的校友账号 | ①学生发起结构化求助（指定专业标签+问题类型）②查看被通知校友的`/notifications` | `help_ticket.status`由OPEN→MATCHED；通知优先投给Tier1候选池（同专业校友/高年级学长，按`40×同专业+15×校友身份+5×年级差(封顶3)+6×同类型采纳数(封顶5)+3×log(累计采纳数)`降序取Top-K）；若Tier1为空则Tier2任一管理员兜底，保证≥1条通知落库 |
| TC-03 | 三段式回答 + 采纳 → AFTER_COMMIT生成知识候选 | 存在一条MATCHED状态求助单 | ①校友按precondition/steps/cautions三段式提交回答②提问者对该回答执行"采纳" | `help_answer.isAdopted=true`；`help_ticket.status`→ADOPTED；事务提交后异步触发`HelpAnswerAdoptedEvent`（`@TransactionalEventListener(phase=AFTER_COMMIT)`），`createFromHelpAdoption`生成`knowledge_entry`（status=CANDIDATE，sourceType/sourceHelpId回填），并回写`help_answer.knowledgeEntryId`；建议轮询/延时校验以体现异步特性 |
| TC-04 | 知识三态评价 | 存在一条PUBLISHED知识条目，两个不同用户账号 | ①用户A评价"有用"②用户A改评"需更新"③用户B评价"已过时" | `knowledge_feedback`按用户+条目落一条可变更记录（同一用户重复评价为切换而非叠加）；条目下三态统计数正确联动更新 |
| TC-05 | 管理员审核通过 / 退回 / 驳回 | 审核队列中存在AUTH_APPLICATION类型PENDING任务（该类型三决定均支持） | 分别对3条待审任务执行APPROVE/RETURN/REJECT | `audit_task.status`分别落APPROVED/RETURNED/REJECTED；`AuthApplicationService.approve/returnForSupplement/reject`被对应调用，`user.authStatus`同步变化。**边界补充用例**：对KNOWLEDGE_ENTRY类型任务执行REJECT，预期抛业务异常"知识候选无独立拒绝语义，仅支持通过/退回"（`KnowledgeEntryAuditHandler`无REJECT分支），验证异常输入处理 |
| TC-06 | 举报处理四种处置 | 存在PENDING举报，分别指向路径卡/知识条目/用户等不同target_type | 分别执行NONE（仅记录不处置）/CONTENT_HIDDEN（隐藏路径卡）/CONTENT_OFFLINE（知识/机会下线）/USER_DISABLED（封禁账号） | `report.status`PENDING→HANDLED（成立）或DISMISSED（不成立）；处置动作码正确校验target_type匹配（如CONTENT_HIDDEN只能作用于ALUMNI_PATH_CARD），越权目标类型应被拒绝 |
| TC-07 | 标签增改停用 | 管理员登录TagManage | ①新增标签②编辑标签名③停用某标签 | `tag`表增删改成功；停用后该标签在HelpCreate等选择器中不可再选，但已引用该标签的历史数据不受影响 |
| TC-08 | 关注 / 取关 + 粉丝数 | 两个已登录账号A、B | ①A对B执行`POST /users/{id}/follow`②再执行`DELETE`取关③自己关注自己 | `user_follow`增删对应记录；`GET /users/{id}/follow-status`正确返回状态；B的粉丝数±1联动；自关注请求被拦截返回业务错误（幂等/越权防护） |
| TC-09 | 私信发送 + 会话聚合 + 未读清零 | A、B互相可见 | ①A向B发送多条消息②B打开`/messages`会话列表③B点开与A的会话 | `message`表写入，`isRead=0`初始未读；会话列表按对端分组仅显示最后一条+未读数`unreadCount`；点开会话后`PATCH .../read`触发未读数清零为0 |
| TC-10 | 查看他人主页 | 目标用户资料公开（`profileVisibility`允许） | 访问`/u/:id`（含未登录访客） | 页面按`GET /users/{id}/public`正确渲染只读信息（关注数/徽章/标签/公开内容）；若`profileVisibility`受限，联系方式等敏感字段不返回 |
| TC-11 | 徽章置顶 / 隐藏 | 用户已拥有多枚徽章 | ①置顶某徽章②隐藏某徽章③查看自己/他人主页效果 | `PATCH /badges/{id}`成功；`pinned=true`的徽章在个人主页优先展示；`hidden=true`的徽章不出现在他人主页`GET /users/{id}/public`返回中，但本人管理面板仍可见 |
| TC-12 | 身份认证申请提交 / 撤回 | 用户`authStatus`未认证 | ①提交认证申请（材料信息）②撤回申请③管理员侧查看队列变化 | `auth_application`写入PENDING记录并对应生成AUTH_APPLICATION类型`audit_task`；撤回后申请状态终止，不再出现在审核队列；重复提交应做幂等/互斥校验 |
| TC-13 | 越权防护（操作他人数据被拒） | A、B为不同普通用户 | A尝试用自己token调用"标记B的通知已读/删除B的私信/编辑B的徽章"等接口，直接改URL中的目标id为B的id | 接口返回403/业务错误，而非误操作成功；服务端逻辑基于`SecurityUtil.currentUserId()`校验资源归属，不完全信任请求体传入的userId |
| TC-14 | 登录态失效 → 401跳登录 | 已登录，token过期或被篡改 | 携带失效token访问任意`requiresAuth`页面/接口 | 响应拦截器捕获401，清除本地token并跳转`/login?redirect=<原路径>`；重新登录后可跳回原页面 |
