- **怎么写**:放"概要设计——前端总体设计"或"系统实现——前端关键技术"。先讲动机(通用组件库难体现双圈品牌调性、样式代码重复率高),再分类列举组件族与职责,最后讲设计令牌如何做到"改一处、全局生效"。可在"总结"作创新点简述。
- **放哪(图/表)**:表【前端 UIKit 组件分类表】(名称/分类/数量/职责);图【UIKit 组件依赖包图】(UML 包图:base → cards/navigation/feedback → 业务页面)。

### 3. 采纳→知识候选 AFTER_COMMIT 异步事件链
- **写什么**:求助单被采纳(ADOPTED)后,通过 `@TransactionalEventListener(phase = AFTER_COMMIT)` 监听 `HelpAnswerAdoptedEvent`,在采纳事务**提交后**异步执行 `createFromHelpAdoption` 生成知识候选(CANDIDATE)并回写关联,避免事务回滚产生脏数据,主流程不被知识生成逻辑拖慢。
- **怎么写**:放"详细设计——核心业务流程设计"。先给业务动机(问答沉淀为可复用知识),用**类图**展示事件/监听器/KnowledgeEntryService 协作关系,用**时序图**展示"采纳请求→事务提交→事件发布→异步监听→生成知识候选"的时间顺序,强调 AFTER_COMMIT 相对同步调用的一致性与解耦优势。
- **放哪(图/表)**:图【采纳生成知识候选时序图】(UML 时序图);图【知识条目状态图】(CANDIDATE→REVIEWING→PUBLISHED/EXPIRED/OFFLINE);代码片段:`@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)`。
- **一句提炼**:"借助 Spring 事务事件监听机制,在采纳事务真正提交后才异步触发知识候选生成,从架构层面杜绝回滚脏数据风险,同时把非核心链路从主流程解耦。"

### 4. 校友路由匹配算法
- **写什么**:`HelpRouteServiceImpl` 整数加权评分:`W_MAJOR=40`(同专业)+`W_ALUMNI_IDENTITY=15`(校友身份)+`W_GRADE_GAP=5×年级差`(封顶3级)+`W_EXPERTISE=6×同问题类型历史被采纳次数`(封顶5次)+`W_TRUST=3×log(累计被采纳数)`(对数信任防头部垄断);候选池 Tier1(同专业+高年级校友,排除本人/已匹配)为空则降级 Tier2(任一管理员兜底),按分降序取 Top-K 通知。
- **怎么写**:放"详细设计——核心算法设计"。先定义问题(如何把求助单精准路由给合适答主),逐项说明权重设计依据,用**活动图**表达算法流程(含"Tier1 是否为空→是否走 Tier2"判断节点),再讲兜底策略如何保证系统鲁棒性。
- **放哪(图/表)**:图【校友路由匹配算法活动图】(UML 活动图);表【路由匹配权重设计表】(权重项/取值/上限/依据);代码片段:评分方法签名。
- **一句提炼**:"路由匹配采用可解释的整数加权评分模型,专业匹配、校友身份、年级梯度、历史专长、对数信任五维度共同决定推送优先级,并以 Tier1/Tier2 两级候选池兜底,保证每条求助单至少获得一次精准或托底通知。"

### 5. 乐观锁 version + 多状态机
- **写什么**:`knowledge_entry.version` 实现乐观锁,解决并发编辑冲突;三个核心业务对象均建模为显式状态机:HelpTicket(OPEN→MATCHED→ANSWERED→ADOPTED→CLOSED)、KnowledgeEntry(CANDIDATE→REVIEWING→PUBLISHED/EXPIRED/OFFLINE)、AuditTask(PENDING→APPROVED/REJECTED/RETURNED/AUTO_APPROVED)。
- **怎么写**:乐观锁放"详细设计——类设计"(结合 MyBatis-Plus `@Version` 说明);状态机放"详细设计——对象状态设计",各给一张 UML 状态图,说明迁移触发条件与不可逆终态设计,体现业务规则被形式化建模而非散落在 if-else 中。
- **放哪(图/表)**:图【求助单状态图】【知识条目状态图】【审核任务状态图】(三张独立编号 UML 状态图);代码片段:`@Version` 字段声明。
- **一句提炼**:"关键实体生命周期显式建模为 UML 状态图并以 status/version 字段落地,配合乐观锁机制,使业务规则可视化、可验证,并避免高并发编辑下的更新覆盖问题。"

### 6. 私信会话聚合 SQL + 全文检索 FULLTEXT ngram
- **写什么**:`/messages/conversations` 按对端用户分组聚合,一条 SQL 取出每个会话最后一条消息+未读数(避免 N+1 查询);知识库检索使用 MySQL FULLTEXT 索引+ngram 分词器,支持中文全文检索,优于 `LIKE %%` 模糊匹配。
- **怎么写**:放"详细设计——数据库设计"或"系统实现——关键 SQL 设计"。私信部分讲清"消息表是明细粒度、消息中心需要会话粒度"的聚合动机及关键写法(GROUP BY 对端ID+取最新一条+未读数统计);检索部分讲清中文全文检索选型原因(MySQL 5.7+ 原生支持 ngram parser,无需外部搜索引擎)。
- **放哪(图/表)**:表【核心表结构简表:message】;代码片段:`CREATE FULLTEXT INDEX ... WITH PARSER ngram`、`MATCH(...) AGAINST(... IN NATURAL LANGUAGE MODE)`、会话聚合 SQL。
- **一句提炼**:"私信模块用一条分组聚合 SQL 在消息明细表之上还原会话粒度视图,知识检索借助 MySQL 原生 FULLTEXT+ngram 分词实现无需引入外部搜索引擎的中文全文检索能力。"

- **写什么**:统一响应体 `Result{code,message,data}`,接口前缀 `/api/v1`,前端 TypeScript 接口定义与后端 DTO 严格对齐;在真实环境中部署 Spring Boot3+MySQL8,完成真实数据入库,验证注册登录/发求助/路由匹配/采纳→生成知识候选/关注/私信等核心业务闭环全链路可用。

- **放哪(图/表)**:表【核心业务闭环联调用例表】(用例名/前置条件/操作步骤/预期结果/实测结果);图【系统集成部署图】(UML 部署图,可选)。
- **一句提炼**:"开发过程中坚持前后端接口契约先行,并完成 Spring Boot3+MySQL8 的部署与数据落库验证,确保从注册登录到路由匹配、知识沉淀、社交互动的核心业务闭环端到端可用。"

## 二、安全性

### 1. JWT(HS384)双令牌无状态鉴权
- **写什么**:accessToken+refreshToken 双令牌;HS384 签名;`Authorization: Bearer` 携带;服务端 `STATELESS` 无状态会话策略。
- **怎么写**:放"详细设计——安全设计"。先讲为何选无状态 JWT(前后端分离、水平扩展友好),再讲双令牌设计动机(短时 access 保安全、长时 refresh 保体验),给出登录→签发→携带访问→续期的时序图。
- **放哪(图/表)**:图【JWT 双令牌认证时序图】(UML 时序图);代码片段:JWT 签发方法(HS384)、SecurityConfig `STATELESS` 配置。
- **一句提炼**:"认证体系采用 HS384 签名的双令牌无状态方案,兼顾接口安全性、用户体验与横向扩展能力。"

### 2. Spring Security + RBAC(4角色×authStatus)+ 方法级 @PreAuthorize(96处)
- **写什么**:4 种角色与 authStatus(认证状态)构成"角色×认证分级"正交权限模型;全项目 96 处 `@PreAuthorize`(如 `@authGuard.isLogin()` / `isVerified()`)做方法级细粒度校验,而非仅 Controller 层粗粒度拦截。
- **怎么写**:放"详细设计——安全设计——权限模型设计"。先给角色与 authStatus 语义,解释正交设计必要性(同角色下认证前后可操作资源不同),再讲自定义 authGuard 封装校验逻辑供 SpEL 复用,最后用 96 处覆盖量作工程量证据。
- **放哪(图/表)**:表/图【角色-认证状态权限矩阵】;图【RBAC 权限校验类图】(UML 类图,User/Role/authGuard 关系);代码片段:`@PreAuthorize("@authGuard.isVerified()")`。
- **一句提炼**:"在 Spring Security 基础上构建'角色×认证状态'正交 RBAC 模型,并以 96 处方法级 @PreAuthorize 实现比控制器级鉴权更精确的访问控制。"

### 3. BCrypt 密码加密
- **写什么**:密码使用 BCrypt 加盐单向哈希存储,数据库不存明文,同一明文多次加密结果不同,抵御彩虹表攻击。
- **怎么写**:放"详细设计——安全设计"小节(可与 JWT/RBAC 合并为"认证安全"),一两句带过即可。
- **放哪(图/表)**:代码片段:`BCryptPasswordEncoder` Bean 声明 + 注册 `encode()` / 登录 `matches()`。
- **一句提炼**:"用户密码经 BCrypt 加盐哈希存储,数据库中不出现明文密码,降低数据泄露场景下的账号安全风险。"

### 4. 越权防护(操作自己数据 + 自关注拦截)
- **写什么**:通知/私信/徽章等接口均以 `SecurityUtil.currentUserId()` 校验"资源属主=当前用户"才允许读写,防止水平越权(IDOR);关注功能显式拦截"自己关注自己"并保证接口幂等。
- **怎么写**:放"详细设计——安全设计——业务层防护"。举 1-2 个具体接口例子(如"标记消息已读"必须校验 receiverId==currentUserId),点明这是不安全直接对象引用(IDOR)的典型防护手段。
- **放哪(图/表)**:表【关键接口越权校验点清单】(接口/校验字段/校验逻辑);代码片段:属主校验分支。
- **一句提炼**:"在通知、私信、徽章等涉及个人数据的接口上统一核对资源属主,阻断跨用户水平越权,并对自关注等边界场景做显式拦截与幂等处理。"

### 5. MyBatis 参数化防注入 + 逻辑删除
- **写什么**:全部数据访问基于 MyBatis(-Plus)参数化查询(`#{}` 占位符),杜绝字符串拼接注入;采用逻辑删除(`deleted` 字段)而非物理删除,自定义 SQL 需手动带 `deleted=0`。
- **怎么写**:放"详细设计——安全设计——数据访问安全"。说明 `#{}` 与 `${}` 区别及项目统一规范,再讲逻辑删除动机(审计/恢复/避免级联删除)及对自定义 SQL 的额外约束(结合 BaseEntity 说明)。
- **放哪(图/表)**:代码片段:Mapper XML `#{}` 参数绑定;`BaseEntity.deleted` 字段声明。
- **一句提炼**:"数据访问层统一使用参数化查询阻断 SQL 注入,并以 BaseEntity 的 deleted 字段实现逻辑删除,兼顾数据安全与可审计、可恢复能力。"

### 6. 隐私可见性控制 + 知识候选隐私预检
- **写什么**:`contactVisibility`/`profileVisibility` 实现联系方式/资料最小可见范围控制;知识候选生成后先做隐私预检(敏感词/真实姓名/联系方式识别),未通过或需复核者进入人工审核(REVIEWING)而非直接发布(PUBLISHED)。
- **怎么写**:放"详细设计——安全设计——隐私保护"。先讲用户侧可见性字段如何在他人主页展示中生效,再讲知识生产链路上的前置隐私过滤,体现"最小暴露+人工兜底"思路,可与知识条目状态图联动说明(标注 CANDIDATE→REVIEWING 迁移条件之一即隐私预检)。
- **放哪(图/表)**:复用图【知识条目状态图】并标注隐私预检位置;代码片段:隐私预检方法(检测敏感词/手机号/邮箱等正则)。
- **一句提炼**:"用户侧以可见性字段实现资料最小化控制,知识生产链路上则在候选进入审核前先做隐私预检,双管齐下降低隐私泄露风险。"

### 7. 统一异常不泄漏堆栈 + 401 拦截
- **写什么**:`GlobalExceptionHandler` 统一捕获异常,对外仅返回标准化 `Result{code,message}`,不泄漏堆栈与内部实现细节;前端 Axios 拦截器捕获 401 后清除本地 token 并跳转登录页,认证前接口只读。
- **怎么写**:放"详细设计——安全设计——异常与容错"。讲清统一异常处理的两个价值:①用户体验一致 ②安全防护(避免堆栈信息辅助攻击者定位漏洞);401 拦截可同节或放"系统实现——前端工程化"。
- **放哪(图/表)**:图【全局异常处理+401拦截协作时序图】(可选,UML 时序图);代码片段:`@RestControllerAdvice` 类声明 + Axios response 拦截器 401 分支。
- **一句提炼**:"后端以 GlobalExceptionHandler 统一收敛异常、只对外暴露标准化错误信息,前端以 Axios 拦截器对 401 做统一清 token+跳登录处理,前后端共同构成完整的异常安全闭环。"
