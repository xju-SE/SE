## 图4-1 业务流程图 → 改标题为/重画为【活动图】

**现状问题**
现图题为"业务流程图"，画法沿用结构化分析(SA)口径（流程框、判定框等），与论文声明的"面向对象方法学、系统建模只用UML"相矛盾——业务流程在OO方法学下应以UML**活动图**（Activity Diagram）表达，继续沿用流程图画法会造成方法论口径自相矛盾。

**具体改动**
- 图题由"图4-1 业务流程图"改为"图4-1 XXX业务活动图"（例如"求助—路由—采纳业务活动图"，具体业务对象取正文该处实际描述的流程）。
- 按UML活动图规范重画：
  - 起止点用实心圆/终止圆（●/◉），不用流程图的"开始/结束"椭圆。
  - 动作节点用圆角矩形，不用直角矩形。
  - 分支用菱形决策节点，分支线上标注 `[条件]`，不用流程图箭头文字判断。
  - 涉及多角色协同时改用泳道版活动图：学生（求助人）泳道 / 系统（路由算法、异步事件）泳道 / 校友-学长（响应者）泳道，必要时加管理员（审核）泳道。
  - 并发环节用同步条（fork/join），不用多分支箭头硬画并发。
- 内容对齐真实闭环，不遗漏也不虚构分支：发求助(OPEN) → 路由匹配通知(MATCHED) → 三段式回答(ANSWERED) → 采纳(ADOPTED) → `@TransactionalEventListener(AFTER_COMMIT)` 异步生成知识候选(CANDIDATE) → 审核 → 发布(PUBLISHED)/关闭(CLOSED)。

**改后校对要点**
- 图中不再含任何结构化流程图/DFD专用符号（平行四边形输入输出、开口矩形数据存储等）。
- 泳道命名与真实角色一致，不出现未定义角色。
- 状态命名与真实状态机严格一致：HelpTicket 为 OPEN→MATCHED→ANSWERED→ADOPTED→CLOSED；KnowledgeEntry 为 CANDIDATE→REVIEWING→PUBLISHED/EXPIRED/OFFLINE，不自造中间状态。
- 全文（含图目录/插图清单）中"业务流程图"字样是否已同步改为"业务活动图"。

## 图6-2 登录时序图

**现状问题**
时序图中调用链使用方法名 `loginByRole(...)`，但真实代码入口为不带 role 参数的 `login(...)`（角色由认证流程内部随用户记录判定，而非作为入参传入），与真实接口签名不符。

**具体改动**
- 将时序图中的调用消息由 `loginByRole(...)` 改为 `login(...)`，参数按真实DTO调整（如 `login(username, password)` 或 `login(LoginRequest)`），不再传入 role 参数。
- 若原图后续步骤依赖"按角色分支返回不同结果"的语义，需调整为：`login` 内部完成用户查询→密码校验(BCrypt)→JWT签发(HS384)，角色信息(role)作为 User 实体自带字段随查询结果返回，而非登录方法输入。

**改后校对要点**
- 时序图中所有涉及登录调用的消息名统一为 `login`，不残留 `loginByRole` 字样（含参数、返回值命名、图中说明文字）。
- 生命线命名规范：前端(Vue组件/Axios) → Controller(AuthController) → Service(UserService/AuthService) → Mapper/DB，与三层架构 `Controller→Service→ServiceImpl→Mapper` 一致。
- JWT签发环节（accessToken/refreshToken）与密码校验环节（BCrypt比对）是否体现，避免遗漏安全相关关键步骤（呼应正文安全性章节）。
- 消息序号、激活条(activation bar)改名后是否仍对齐，无错位。
