## 二、结构冲突的最终裁决（对齐 schema.sql）

### R1 档案表合并为单一 schema（原 M1/M2 双设计）
`student_profile` / `alumni_profile` 各只有一份定义，专业一律用 **tag 外键**：
- 专业：`major_tag_id BIGINT FK→tag.id`（弃用 M1 的 `major` 自由文本）
- 入学年份：`enroll_year SMALLINT`（弃用 `admission_year`）
- 年级：`grade_level TINYINT(1..10)`（弃用 `grade` 文本，系统按 enroll_year 每年重算）
- 毕业年份：`grad_year SMALLINT`
- 长度统一：`student_no VARCHAR(30)`、`college VARCHAR(100)`
- 身份字段 `real_name` 保留在档案表，**不进入 PUBLIC 可见性**（防泄露认证材料）
- 认证终审回写时，M1 增加 `major 文本 → major_tag_id` 解析步骤（调 `TagQueryService`）

### R2 时间线三表单一 owner = M6
`timeline_template / timeline_node / timeline_node_ref` 以 M6 字段为准（`title/stage/suggested_time/order_no/importance`）；M7 只提供 `/api/v1/admin/timeline-*` 维护路由，**Controller 复用 M6 的 Service Bean**，不另建 schema。枚举 `route_type` 统一 `UNDECIDED/POSTGRAD/EMPLOY/COMPETITION/CIVIL`（弃用 M7 的 EMPLOYMENT/CIVIL_SERVICE）。

### R3 采纳→候选链（M4↔M3）
- 方法统一：`KnowledgeEntryService.createFromHelpAdoption(helpTicketId, helpAnswerId, authorId)`（M3 owner，自读回答正文）
- 事务：**异步**——M4 在 `@TransactionalEventListener(AFTER_COMMIT)` 内调用，独立事务，失败走补偿；采纳不因下游回滚
- 补列：`help_answer` 增 `knowledge_entry_id BIGINT NULL FK→knowledge_entry.id`（回写采纳生成的候选 ID）
- `audit_task.review_kind` 增枚举 `NEW_FROM_HELP`

### R4 通知为全局模块（补大缺口）
新增全局 `notification` 模块：`NotificationService.send(userId, type, content, refType, refId)`；提供 P17 通知中心的列表/标记已读接口。类型 `HELP_MATCH/ADOPT/AUDIT_RESULT/SYSTEM`。核心闭环的"路由通知"依赖它，故纳入迭代一。

### R5 候选池查询（补缺口 C-C）
M2 新增 `listVerifiedUsersByMajor(role, majorTagId, college?, minGradeLevel?)`，供 M4 路由匹配构建候选校友池。（本期 M4 先用 M1 用户表 + profile 直接查，M2 未编码前用同等 SQL 兜底。）

### R6 命名/错误码统一
- 标签只读查询参数统一 `?tagType=`
- 乐观锁错误走 `3xxxx`（业务），权限走 `1xxxx`；弃用 M3 的 `40301/40901`
- 采纳入口服务名 `HelpAnswerService.adopt()`
- 读用户成长标签走 `StudentProfileService.listUserTags`
