# M4 结构化求助 实现说明与画图指引

> 对应代码目录：`backend/src/main/java/com/xju/sem/module/help/`
> 对应设计基线：`docs/design/04_M4_结构化求助_详细设计.md`（下称"04文档"），并以
> `backend/src/main/resources/schema.sql` 的实际列为准（精确列以 schema.sql 为准）做了对齐与必要简化，
> 差异集中见 §4「假设与简化」。
> 本模块是全系统"核心闭环/系统灵魂"的枢纽：**发求助 → 路由通知 → 回答 → 采纳 → 生成知识候选 → 审核入库**。

---

## 1. 模块功能说明（做什么、核心流程，白话）

M4 管理一条求助的完整生命周期，围绕四张表 `help_ticket / help_answer / help_followup / help_route` 展开，
做五件事：

1. **发布结构化求助单**：已认证的学生（主场景）/校友/管理员按"专业 / 年级 / 问题类型 / 目标方向"发起求助。
   专业与年级不由用户填，而是发布时从本人档案（M1/M2）**只读快照**写入——学生带在读年级，校友/管理员无在读年级
   （`grade_level` 置空）。求助单落 `OPEN` 态。
2. **★求助-校友路由匹配（系统灵魂算法）**：求助单一提交，系统立刻（事务提交后异步、毫秒级）执行路由匹配——
   构建候选校友池（同专业已认证校友 + 同专业高年级学长），按"专业相同 / 校友身份 / 年级差 / 该类问题历史被采纳次数 /
   累计被采纳信任度"打分，排序取 TopK（`sem.help.route-top-k`），写入 `help_route` 并逐条发站内通知（`HELP_MATCH`）。
   候选池不足时逐级放宽（全平台校友 → 管理员兜底），**保证任何一条求助至少触发一次匹配通知**；求助单随之 `OPEN→MATCHED`。
3. **模板化回答**：符合条件的人以三段式（前提 `precondition` / 步骤 `steps[]` / 注意事项 `cautions`）作答；
   首条回答把求助单推到 `ANSWERED`；同一人对同一单只保留一条回答（可编辑，不可重复提交）。
4. **限次追问**：求助人可对求助单发起追问，`followup_count` 每次 +1，达到 `sem.help.followup-limit`（默认 3）即拒绝
   （抛 `LIMIT_EXCEEDED`）；回答人回复不计次数。
5. **采纳与闭环**：求助人从多条回答里采纳一条最佳回答——置 `is_adopted=1`、求助单 `ANSWERED→ADOPTED`，并发出
   `HelpAnswerAdoptedEvent`。事务提交后由监听器调 M3 `KnowledgeEntryService.createFromHelpAdoption` 生成知识候选，
   再把候选 id 回写到 `help_answer.knowledge_entry_id`——一条被采纳的口头经验就此沉淀为待审核的知识资产。

求助单状态机全程为 `OPEN→MATCHED→ANSWERED→ADOPTED→CLOSED`；两个定时任务负责"长期零应答重试放宽路由"与
"超时/采纳宽限期自动关闭归档"。

**边界（不做的事）**：不做知识候选的审核发布（M3/M7）；不做标签体系维护（M7）；不做认证/角色鉴权本身（M1，仅调用
`isVerified`/`getRole`）；不做通知中心呈现（仅作为 `notification` 生产者调 `NotificationService.send` 写入）。

## 4. 关键实现点

### 4.1 发布快照（§6.1）
`createTicket` 通过 `UserService.getById(askerId)` 拿到发布人 `majorTagId/gradeLevel/role`，只读快照进 `help_ticket`。
学生带在读 `grade_level`；校友/管理员 `grade_level=null`——这一个 null 顺带成了后续路由的"分支开关"（见 4.3），
不需要额外存"发布人角色快照"列。发布前置 `authStatus=VERIFIED` 由 Controller `@PreAuthorize("@authGuard.isVerified()")`
与 Service 双重校验。

### 4.2 状态机与并发（CAS 取代乐观锁）
schema 的 `help_ticket` **没有 version 列**，因此所有状态流转都用"带 `WHERE status=?` 条件的 CAS UPDATE"实现
（`HelpTicketMapper.casStatus/markAnswered/closeTicket`）：受影响行数为 0 即判定"前置状态不满足或并发被他人先改"，
转 `OPTIMISTIC_LOCK/STATE_CONFLICT`。这套 CAS 同时覆盖了"求助人手动关闭"与"定时任务自动关闭/采纳"的竞态双写——
自动关闭任务用"按读到的原状态 CAS"（`casStatus(id, oldStatus, CLOSED)`），若期间状态被用户改动则本次跳过，
与 04 文档要求的乐观锁语义等价。

### 4.3 ★路由匹配算法（§6.2，系统灵魂）
- **候选池分层放宽**：tier1 同专业校友 +（`grade_level != null` 才查）同专业高年级学长 → 不足 3 人加 tier2 全平台校友 →
  仍为空加 tier3 任一管理员。三层保证"任何求助至少有一条 `help_route` 与一次 `HELP_MATCH` 通知"这一验收红线恒成立。
  `grade_level == null`（校友/管理员发起）时天然跳过学长子池，自动退化为"纯校友池"，正好复现 04 文档按发布人角色分支的意图。
- **打分维度（Service 层常量，非魔法数）**：`W_MAJOR=40 / W_ALUMNI_IDENTITY=15 / W_GRADE_GAP=5(封顶3级) /
  W_EXPERTISE=6(封顶5次) / W_TRUST=3`。专长与信任两维直接读**本模块** `help_answer.countAdopted`，不依赖 M2 缓存计数，
  信任维用 `round(W_TRUST·ln(1+累计采纳))` 对数缩放防头部垄断。
- **打分伪代码**：
  ```
  for u in pool:
      s = 0; reasons = []
      if u.major == ticket.major:  s += 40;  reasons += "专业相同"
      if u.role == ALUMNI:         s += 15;  reasons += "校友身份"
      elif u.role == STUDENT and ticket.grade != null and u.grade != null:
          gap = u.grade - ticket.grade
          if gap > 0: s += min(gap,3)*5; reasons += "高年级学长学姐"
      if ticket.qType != null:
          same = countAdopted(u.id, ticket.qType)      // 读 help_answer
          if same > 0: s += min(same,5)*6; reasons += "该类问题曾被采纳"+same+"次"
      total = countAdopted(u.id, null)
      if total > 0: s += round(3*ln(1+total))
  ranked = sortDesc(pool, by=s);  topK = ranked.take(sem.help.route-top-k)
  for (rank,u) in topK: insert help_route(NOTIFIED, score=s); notify(HELP_MATCH)
  if topK not empty: casStatus(OPEN→MATCHED)
  ```
- **去重与重试**：`help_route` 唯一约束 `uk_ticket_user(ticket_id, matched_user_id)` 兜底防重复通知同一人；
  每轮先查 `listMatchedUserIds` 放进排除集，`HelpRouteRetryJob` 靠"排除已匹配者 + 分层放宽"实现"分批升级"，
  无需 schema 里的 `batch_no` 列。TopK 由 `sem.help.route-top-k` 配置（默认 20），排序用"分数降序、userId 升序"稳定排序。

### 4.4 事件解耦与事务边界
- **发布 → 路由**：`createTicket` 事务内 `publishEvent(HelpTicketCreatedEvent)`，`HelpTicketCreatedListener` 以
  `@TransactionalEventListener(AFTER_COMMIT)` 监听后调 `routeHelpTicket`——把路由从发布主链路解耦（不拖慢响应），
  并保证路由读到的是已提交的求助单行。
- **采纳 → M3 候选**：`adopt` 的 `@Transactional` 只覆盖"置 `is_adopted` + CAS 状态 + 回写路由响应"这一最小核心写；
  生成候选、回写 `knowledge_entry_id`、采纳通知统一交给 `HelpAnswerAdoptedListener`（AFTER_COMMIT）异步完成，
  任一下游失败**只记日志走补偿、不回滚已成功的采纳动作**（下游故障隔离，符合低耦合）。
- **刻意打断循环依赖**：M3 `KnowledgeEntryService` 依赖 M4 `HelpAnswerService`（读回答正文），若采纳逻辑又直接依赖
  `KnowledgeEntryService` 会构成构造器循环依赖。故把"采纳→M3"放进**独立监听器** `HelpAnswerAdoptedListener`，
  `HelpAnswerServiceImpl` 本身不引用 `KnowledgeEntryService`，依赖图无环。监听器不加 `@Transactional`：
  M3 的 `createFromHelpAdoption` 自身是 REQUIRED 事务会独立开启并提交，回写用单条 `updateById` 自动提交。

### 4.5 限次追问（§6.3）
求助人追问用**原子自增判定** `UPDATE ... SET followup_count=followup_count+1 WHERE id=? AND followup_count<limit`，
受影响行数为 0 即已达上限 → `LIMIT_EXCEEDED`，避免并发超发。回答人回复不计次数。追问按**求助单**组织线程
（schema `help_followup` 无 `target_answer_id`），发送方是求助人则通知全部回答人、是回答人则通知求助人。

### 4.6 列表统计卡与详情操作位
- 列表默认按当前用户 `major_tag_id` 过滤（"本专业高频"，非信息流表格），支持问题类型/状态筛选与
  `LATEST/NEARLY_TIMEOUT/UNANSWERED_FIRST` 排序（后者用 `ORDER BY FIELD(status,...)`）；统计卡三项
  （待解决数/已解决数/平均响应时长）全部 SQL 级聚合，平均响应时长 = 求助创建到首条回答的 `TIMESTAMPDIFF(HOUR)` 均值。
- 详情返回 `myActions{canAnswer,canFollowUp,canAdopt}` 作为前端入口提示（写接口仍二次校验），越权（改 URL 里的 id）
  在 Service 层按"求助人本人 / 回答人本人"属主校验拦截。

### 4.7 假设与简化（相对 04 文档，以 schema.sql 为准）
1. **help_ticket 无 version/close_reason/adopted_answer_id 列**：状态并发改用 CAS（4.2）；`closeReason` 仅日志留痕不入库；
   已采纳回答通过 `help_answer.is_adopted=1` 唯一定位，不冗余存 `adopted_answer_id`。
2. **help_answer 无 uk_ticket_responder、无 responder_role_snapshot**：一人一单一答由 Service 应用级去重；回答人角色/校友徽标
   在出参时经 `UserService.getBrief` 实时取，不落快照列。`steps` 列为 VARCHAR，以 JSON 数组文本存储（JacksonTypeHandler）。
3. **help_followup 极简（无 target_answer_id/sender_role/follow_up_seq/updated_at）**：追问改为**按求助单**的单线程模型，
   限次口径落在 `help_ticket.followup_count`（仅求助人计数），与 04 文档"按回答分线程 + 每回答限次"相比是随 schema 的简化。
4. **help_route 极简（无 batch_no/rank_no/route_type/match_reasons/deleted）**：不存匹配理由与批次号——理由折进通知文案，
   批次升级靠"排除已匹配者 + 候选池分层"实现；`match_score` 为 INT（对分数取整）。
5. **target_direction 为自由文本而非标签外键**：本期**不纳入路由打分维度**（避免越界查 user_tag/tag 表），
   与任务书列出的打分维度（专业/问题类型/年级）一致；未来标签化后可补 `W_DIRECTION`（GROWTH 命中）。
6. **候选池直连他模块表**：`HelpRouteMapper` 直连 `user + student_profile/alumni_profile` 查候选池，是任务书授权的本期做法，
   已在类注释标注未来由 `M2.listVerifiedUsersByMajor` 提供后删除这些 SQL、改走 Service 契约。
7. **未做的对外方法**：`getSummary/hideTicket/restoreTicket/getMyCollegeOpenTicketStats` 等 M7/首页占位契约本期未实现
   （无调用方），留待相应模块落地时按需补齐；统计卡逻辑已具备，届时可直接复用。
8. **依赖前提**：`NotificationService`（`com.xju.sem.module.notification.service`）由通知模块提供、`KnowledgeEntryService`
   由 M3 提供——二者均已被现有 M1/M3 代码引用，属既有跨模块契约；所有对外通知调用都包了 try/catch，发送失败不阻断主流程。
