## 1. 模块功能说明（做什么、核心流程）

本次交付是 M7 治理能力的"收尾"，围绕四类新增职责：

1. **标签体系维护**：`tag` 是全局基础表（字段定义归 M2），但按 02/07 详细设计的既定分工，
   其增删改端点与 `TagMapper` 物理落在本模块。ADMIN 在管理后台维护 MAJOR/GRADE/INDUSTRY/
   INTEREST/GROWTH/QUESTION_TYPE 六类标签的树形结构，停用前可看到"被 N 处业务记录引用"的
   只读提示（跨表 COUNT 聚合，不阻断操作）；对外暴露 `TagQueryService`（只读 list/getById/
   resolveMajorTag）供 M1/M2/M4/M5/M6 等全模块按名称解析专业标签或渲染下拉框。
2. **举报受理闭环**：登录用户（含未认证）可对求助单/知识条目/校友路径卡/机会/账号提交举报
   （同一人对同一目标的重复待处理举报自动合并说明，不新建记录）；ADMIN 在治理端队列里逐条
   判定"成立/不成立"，成立时按目标类型分发调用对应模块已暴露的下架/下线/封禁方法——本模块
   自己不实现任何一种内容治理动作，只做"举报单生命周期 + 路由转发"。
3. **贡献者认证申请与审核**：已认证校友可提交荣誉证明申请"贡献者"标识，材料随一条
   `audit_task`(target_type=CONTRIBUTOR_CERT) 流入统一审核队列，与认证申请、知识候选、
   机会终审共用同一套"分发到 Handler"机制；ADMIN 审核通过后由 M2 `AlumniProfileService.
   grantContributorBadge` 写入徽章，本模块只管流程，不碰 `alumni_profile` 表。
4. **机会终审接入统一队列**：M5 机会模块落地并开始发布 `OpportunitySubmittedEvent` 后，本次
   补充监听该事件建 `audit_task`(target_type=OPPORTUNITY)，并新增 `OpportunityAuditHandler`
   完成"通过/拒绝"终审转发——07 文档设计的四类审核目标（认证申请/知识候选/机会/贡献者认证）
   至此全部接入同一条统一审核队列，无需再为机会单独开一套审核界面。
5. **运营数据统计看板**：聚合用户认证通过率、知识库候选流水线分布、机会/组队规模、举报处理
   时长中位数、贡献者数量、审核吞吐量趋势（含 §6.5 高峰期理论吞吐量 PDAT 估算线）产出只读
   统计视图，全程只调用各模块已暴露的分页方法读 `total` 字段拼装，不为统计目的新增任何
   跨模块契约方法（07 文档 §6.6 明确的克制原则）。

**本次刻意不做/交由既有实现的事**：内容本身的下架/下线/封禁规则（各归属模块）；时间线模板/
节点/引用的维护端点与数据访问层（已由 M6 并行交付，见 §4.2，本次不重复建设）；求助单举报的
"强制关闭"能力（M4 尚未开放管理员入口，本次记录缺口而非假装实现，见 §4.3）。

## 4. 关键实现点

### 4.1 折叠式列复用：`auto_precheck`/`decision_note`/`reason`/`handle_note` 的"一列多义"模式
schema.sql 为课程规模做了大量列精简（`audit_task` 无 `payload`/`checklist_result` 独立列，
`report` 无 `reason_type`/`handle_action` 独立列）。本次延续 07_M7_实现说明.md 已确立的处理
风格——**通用文本列按 target_type/场景语义复用**，而非为每种语义新增列：

- `audit_task.auto_precheck`（VARCHAR(500)，本为 KNOWLEDGE_ENTRY 预检结果预留）在
  CONTRIBUTOR_CERT 场景下改存 `ContributorCertPayload{honorCertUrl, note}` 的 JSON 文本
  （`ContributorCertServiceImpl.apply` 写入，`ContributorCertAuditHandler`/
  `AuditTaskServiceImpl.getById` 按需解析）。
- `report.reason`/`report.handle_note` 统一用 `"[CODE] 文本"` 格式折叠"分类码 + 自由文本"
  （`ReportServiceImpl#fold`/`#splitCode`），CODE 取 `ReportReasonType`/`ReportHandleAction`
  枚举名，解析失败（如历史脏数据）时整体归入文本部分，不抛异常。

这一模式的取舍：牺牲了"列本身语义单一"的整洁性，换来"不需要为每个精简掉的字段申请加列"，
与 07_M7_实现说明.md §4.5 的 `auto_precheck` 处理如出一辙，属同一工程判断在新场景下的延续。

### 4.2 时间线模板/节点维护：本次未新增代码，已由 M6 并行交付
07 详细设计 §3.4-3.6 将 `timeline_template`/`timeline_node`/`timeline_node_ref` 的**治理端**
字段设计与维护规则先行写在 M7 文档里（"名义归属 M6，本处为治理端 CRUD 预留字段设计"）。本次
任务书据此要求补一个 `TimelineAdminController`。但开始编码前 Read 现有代码时发现：**M6 侧已
在本轮并行交付了完整的维护端实现**——`TimelineTemplateController`（`/api/v1/timeline-templates`，
增删改查/发布/下线，`@PreAuthorize("hasRole('ADMIN')")`）、`TimelineNodeController`
（`/api/v1/timeline-nodes`，节点编辑/删除/引用覆盖式更新 `PUT .../refs`）与对应
`TimelineTemplateService`/`TimelineNodeService`/`TimelineNodeRefService` 三个接口 + Mapper，
字段与状态机（`DRAFT/PUBLISHED/OFFLINE`，状态 CAS，因 schema 无 `version` 列故未采用乐观锁）
均与 07 文档 §3.4-6.8 的设计意图一致，且 `TimelineNodeRefService.countByRef(String,Long)`
已按地基契约"M6(提供,供M3的P09计数)"签名原样暴露。

**本次的处理**：不重复建设 `/api/v1/admin/timeline-*` 这一套平行路由——两套路由指向同一份
业务规则会造成维护双份代码、且容易出现"两个入口校验不一致"的风险，与"不得破坏或重复已有
类"的原则相悖。改为在本文档中记录对接情况：P18 管理后台"Tab③ 时间线模板维护"区可直接调用
M6 已交付的 `/api/v1/timeline-templates`/`/api/v1/timeline-nodes` 两组接口，功能范围
（模板 CRUD/发布/下线、节点 CRUD、引用覆盖式更新）已完整覆盖 07 文档 §5(b) 对应条目，仅 URL
前缀由 `/api/v1/admin/timeline-*` 变为不带 `admin` 前缀的资源路径（鉴权注解已是
`hasRole('ADMIN')`，实际效果等价）。

### 4.3 举报处理的已知缺口：HELP_TICKET 无法自动下线
`HelpTicketService` 目前只暴露 `close(ticketId, operatorId, reason)`，其内部严格校验
`operatorId.equals(ticket.getAskerId())`（求助人本人专属，无 ADMIN 放行分支）——这与 M4 自己
的 04 实现说明 §4.7 第 7 条"`hideTicket`/`restoreTicket` 本期未实现，留待相应模块落地时按需
补齐"完全吻合：本模块正是那个"相应模块"，但受任务边界约束（只写 `module/admin`，不改
`module/help`），选择**不假装调用会成功**——`ReportServiceImpl#requireActionMatch` 对
`HELP_TICKET` 的任何非 `NONE` 处理动作直接抛 `AdminErrorCode.REPORT_ACTION_UNSUPPORTED`，
仅允许"成立但不自动处置（NONE，需人工线下跟进）"。待 M4 补充管理员强制关闭方法后，只需在
`ReportServiceImpl#dispatch`/`#requireActionMatch` 追加一个分支即可接通，不影响本次已交付的
其余四类分发逻辑。

### 4.4 CONTRIBUTOR_CERT 终审：不改动既有 `AuditTargetHandler` 接口签名
`AuditTargetHandler.handle(Long targetId, Long reviewerId, AuditDecision decision, String
comment)` 只传 `target_id`（=申请人 userId），不携带 `audit_task.id`，因此
`ContributorCertAuditHandler` 无法直接从入参拿到 `auto_precheck` 里的 `honorCertUrl`。
两个可选方案：①扩大接口签名传入完整 `AuditTask`（影响已有 `AuthApplicationAuditHandler`/
`KnowledgeEntryAuditHandler` 两个既有实现，属于"改动既有交付物"）；②在 Handler 内部直接注入
`AuditTaskMapper`（同模块内部访问，非跨模块违规）反查
`target_type=CONTRIBUTOR_CERT AND target_id=userId ORDER BY id DESC LIMIT 1`。选择方案②——
`ContributorCertServiceImpl.apply` 已保证同一用户同时最多一条 `PENDING` 申请（重复申请抛
`30709`），故终审时刻"按 target_id 取最新一条"能唯一定位到正在被处理的任务，不改动既有接口
与两个已交付 Handler 实现。

### 4.5 事务边界与既有模式保持一致
`ReportServiceImpl.handle()` 与 `AuditTaskServiceImpl.decide()` 采用同一强一致策略：状态 CAS
更新与目标模块治理动作复用同一物理事务（`@Transactional`，`propagation=REQUIRED`），任一环节
失败整体回滚，不出现"举报已标记成立但内容其实未下线"的中间态；`ContributorCertServiceImpl.
apply()`/`AuditTaskEventListener.onOpportunitySubmitted()` 分别延续既有"提交侧"简化事务与
"AFTER_COMMIT 异步解耦"两种既定模式，未引入第三种一致性策略。

### 4.6 标签使用计数：跨表只读聚合的"受控例外"
`TagMapper.countUsage` 直接 `UNION`/累加 `user_tag`/`student_profile`/`alumni_profile`/
`alumni_path_card`/`help_ticket`/`timeline_template` 六张他模块表的 `tag_id` 外键列计数，
不经各模块 Service。这与 M1 现有 `MajorTagMapper`（只读反查 `tag` 表解析 major_tag_id）同一
"受控例外"性质——纯只读聚合、不写、不越权，且 07 详细设计 §6.7 本身就要求这一算法形态
（"聚合各标签被 user_tag/help_ticket 等引用的次数，只读聚合，不改写来源表"）。`M1` 现有的
`MajorTagResolver`/`MajorTagMapper`（内部直连 `tag` 表解析专业名）与本次新增的
`TagQueryService.resolveMajorTag` 并行存在——未改造 M1 既有实现以避免破坏其已交付代码，
未来 M1 侧可自行选择迁移到本契约。

### 4.7 运营统计的已知缺口：`helpResolveRate` 恒为 null
07 详细设计 §6.6 给出的算法假设 `HelpTicketService` 能做全站口径的解决率统计，但现有
`listTickets(query, viewerId)` 在 `majorTagId` 为空时会退化为"取当前登录人本人专业"
（M4 04 实现说明 §4.6），ADMIN 账号没有 `student_profile`，不适用于全站聚合；而 §6.6 又
明确"不为统计目的要求其他模块新增专用计数接口"。二者相叠加，使该项在本次不越权、不新增
契约的前提下无法产出——`OperationOverviewDTO.helpResolveRate` 字段保留但恒为 `null`，字段
注释已写明原因，留待 M4/M7 协商暴露一个不依赖登录态专业上下文的全局统计只读方法后再补齐。

### 4.8 PDAT（高峰期最大日审核吞吐量）估算实现
按 07 详细设计 §6.5 公式 `Σ_type(admin_online_count × daily_available_minutes × 60 ÷
avg_decision_seconds[type])` 逐类型求和实现（`OperationStatsServiceImpl#estimatePeakDailyThroughput`），
四类基准秒数与人力参数均做成 `@Value` 可配置项（默认值取 07 文档给定的基准表），供答辩现场
调参展示；与 07 文档"典型算例"叙述中使用的"四类加权平均秒数"简化写法（≈41 条/日）是同一
模型的两种不同展示方式，本实现采用公式本身的逐类型求和，数值会与叙述性例子的近似值略有出入，
属预期内的正常差异。
