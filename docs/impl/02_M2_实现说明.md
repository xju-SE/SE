# 02 M2 成长画像与校友路径 实现说明

> 代码位置：`backend/src/main/java/com/xju/sem/module/profile/`
> 对齐：详细设计 `docs/design/02_M2_成长画像与校友路径_详细设计.md`，列口径以 `backend/src/main/resources/schema.sql` 为准（schema 与详细设计有出入时以 schema 为准，差异见 §4）。

## 1 模块功能说明

M2 负责两类主体的结构化画像与"去向"数据资产，以及基于它们的可见性、统计与推荐能力：

| 功能 | 入口 | 说明 |
|---|---|---|
| ① 在校生画像维护 | `StudentProfileService.updateProfile` | GPA（校验 `[0,gpaScale]`，`gpaScale∈{4,5}`）、目标城市、目标行业（须 `tag_type=INDUSTRY`）、简介、头像；专业/年级/学号为认证结果只读不改 |
| ② 成长标签维护 | `UserTagService.updateMyTags` | 覆盖式写 `user_tag`；去重、上限 10、类型须 `INTEREST/GROWTH`；`user_tag` 无 `deleted` 列，采用"物理删旧+插新" |
| ③ 校友路径卡 CRUD（去向分支） | `AlumniPathCardService.create/update` | `EMPLOY` 展开城市/行业/公司/岗位；`POSTGRAD` 展开目标院校/专业/初试成绩/复试/备考时长；其余去向仅通用字段 |
| ④ 发布/撤回状态机 | `publish/withdraw/hidePathCardByReport/restorePathCard` | `DRAFT/PUBLISHED/HIDDEN`，`version` 乐观锁并发控制 |
| ⑤ 字段级可见性 | `getVisibility/updateVisibility` + `PathCardVisibilityResolver` | `path_visibility` 分组级 `SELF/SAME_MAJOR/PUBLIC`；`real_name` 不落路径卡表，任何脱敏 DTO 都不携带真名 |
| ⑥ 浏览列表 + 详情（按 viewer 脱敏） | `pageList/getDetail/getVisiblePathCard` | 仅 `PUBLISHED` 入列；逐卡按访问者身份计算可见字段分组 |
| ⑦ 按专业去向统计 + 样本门槛 | `MajorDestinationStatsService.getStats` | `total < sem.knowledge.stat-min-sample(默认20)` 仅返回"样本不足，仅供参考"、隐藏百分比；二级维度 k-匿名（阈值 5）合并 `OTHER` |
| ⑧ 路径推荐规则引擎 | `PathRecommendationService.recommend` | 输入专业/年级/GPA区间/兴趣/目标城市行业 → 候选池 → 加权打分 → 多样性去重 → 分去向 TopN + 匹配校友 |
| ⑨ 对外契约 | 见下 | 供 M4/M6/M7/首页仪表盘调用 |

**对外暴露的跨模块契约（调用方按签名调用）**：
- `ProfileQueryService.listVerifiedUsersByMajor(String role, Long majorTagId, String college, Integer minGradeLevel) → List<VerifiedUserDTO>`（供 M4 路由候选池 / M6 同专业参考人 / M7 贡献者审核筛选）
- `AlumniPathCardService.getVisiblePathCard(Long cardId, Long viewerId) → VisiblePathCardDTO`（供 M6 节点引用，脱敏，不可展示返回 null）
- `AlumniPathCardService.existsPathCard(Long id) → boolean`、`hasMajorTag(Long userId, Long majorTagId) → boolean`
- `StudentProfileService.getProfile(Long userId) → StudentProfileDTO`、`listUserTags(Long userId) → List<TagDTO>`
- `AlumniProfileService.getBrief(Long userId) → AlumniBriefDTO`、`grantContributorBadge(Long userId, String honorCertUrl)`、`incrementHelpedCount/incrementAdoptedCount`（供 M4 采纳事件）
- `AlumniPathCardService.hidePathCardByReport/restorePathCard`（供 M7 治理）

**依赖的既有能力（只调 Service 契约）**：`UserService.getBrief/isVerified`（M1）。

    │   / AlumniPathCardService / MajorDestinationStatsService / PathRecommendationService

分层职责：Controller 只做入参校验 + 鉴权注解 + 转发；Service 承载全部业务规则；Mapper 仅表访问（聚合用注解 SQL 在 DB 侧完成分组，不"读全表再内存分组"）。

## 4 关键实现点

1. **schema 与详细设计的差异对齐（以 schema 为准）**：
   - 路径卡列名用 schema：`grad_stage(BACHELOR/MASTER/PHD)`、`grad_gpa`（无 `gpa_scale`）、`city/industry_tag_id/company/position`、`target_school/target_major/exam_score(VARCHAR100 自由文本)/interview_exp(VARCHAR1000)/prep_months`、`advice(VARCHAR2000)`、`version`。
   - schema 无 `postgrad_admission_type`/初试成绩 JSON/`prep_materials`，故 `POSTGRAD` 分支必填收敛为"目标院校 + 目标专业"，`exam_score` 为可空自由文本；备考时长校验 `[0,60]`。毕业 GPA 无满分制列，按最大满分 5 做上界。
   - `destination_type` 以 schema 五值为核心，枚举补充 `ENTREPRENEUR/FLEXIBLE`（仍以枚举名字符串落库，只有 EMPLOY/POSTGRAD 展开分支）。

2. **M2↔M1 的唯一受控接缝 `IdentityProfileSupport`**：`student_profile/alumni_profile` 的 entity+BaseMapper 由 M1 认证流程先建；为避免**重复**其 entity（两个 `@TableName` 指向同一物理表），M2 复用 M1 的 `StudentProfileMapper/AlumniProfileMapper`（均为无业务逻辑的裸 BaseMapper，复用不绕过任何封装），并把全部此类表访问收敛到本类一处，便于 M1 将来下沉 Repository 契约时集中替换。M1 内部亦有同类受控例外（`MajorTagResolver` 只读 M2 的 `tag` 表）。跨读账号态一律走 `UserService` 契约（`getBrief`）。

3. **`tag` 只读访问 `TagReadMapper`**：系统暂无统一 `TagQueryService`（M7 剩余）。本模块自建只读 `Tag` entity + `TagReadMapper`（绝不写 tag 表），用于目标行业/专业类型校验与标签名回填；待 M7 提供 `TagQueryService` 后可整体替换 `ProfileTagSupport` 内部实现，不影响上层。

4. **无 `deleted` 列的两张表不继承 `BaseEntity`**：`user_tag`、`path_visibility` 在 schema 只有 4~5 列且无 `deleted/updated_at`；若继承 `BaseEntity`，其 `@TableLogic deleted` 会向 WHERE 注入不存在的列导致 SQL 失败。故二者为独立 `@TableId` 实体，覆盖式更新采用物理删/插。

5. **乐观锁与"全量覆盖"更新的坑**：路径卡 `@Version` 由全局 `OptimisticLockerInnerInterceptor` 生效；`updateById` 受影响行为 0 时手动抛 `30203`（不依赖框架抛异常）。切换去向类型时 §6.1 会把非当前分支字段清空为 `null`，但 MyBatis-Plus `updateById` 默认 `NOT_NULL` 策略会**跳过 null 字段**导致脏数据残留——故对所有可空分支/通用列显式标注 `@TableField(updateStrategy = FieldStrategy.IGNORED)`，让编辑的全量覆盖语义真正写入 null；必填列保持默认策略。

6. **脱敏引擎纯函数化 `PathCardVisibilityResolver`**：只依赖入参（卡片 + 可见性配置 + 访问者关系判定），不做 DB 访问，便于列表批量脱敏复用与单测。列表页批量加载可见性（一次 `in` 查询分组）+ 属主摘要缓存，避免逐卡 N+1 的可见性查询。`real_name` 不在路径卡表内，任何对外 DTO 均不携带真名；`ownerNickname` 用登录名。

7. **统计口径与隐私**：聚合基于全部 `status=PUBLISHED` 记录、与字段可见性无关（可见性只控展示给谁）；一级门槛 `sem.knowledge.stat-min-sample`（缺省 20，`@Value` 注入，无需改配置即可跑）不足时不返回任何计数/百分比；二级维度 k-匿名阈值 5，小桶合并 `OTHER`，行业桶回填标签名。

8. **推荐引擎为纯读、结果不落库**：软性加权不做硬性剔除（避免空结果）；冷启动因 schema 无 college 列，"同学院跨专业"简化为"全平台已发布卡限量"；深造意向 `containsInterestSignal` 用兴趣标签名关键词启发式（考研/深造/读研/读博/升学/保研）；jaccard 用校友的 `INTEREST` 自选标签集合，带按属主缓存。

9. **鉴权**：画像/推荐 `@authGuard.isVerified() and hasRole('STUDENT')`；路径卡编辑 `hasRole('ALUMNI')`；成长标签 `hasAnyRole('STUDENT','ALUMNI')`；浏览/统计/详情 `@authGuard.isVerified()`（GUEST 不可访问，对齐"看校友隐私字段需认证"）；下架/复核 `hasRole('ADMIN')`。所有 `/{id}` 编辑在 Service 层二次校验属主（`resource.userId==current or ADMIN`），防改 URL 中 `id` 的水平越权。

10. **定时任务**：`GradeLevelRecalcJob` 每年 9 月 1 日按 `enroll_year` 重算 `grade_level`（连续学年近似 `clamp(year-enrollYear+1,1,10)`，因 `student_profile` 无学历层次列无法精确分段，数值仅用于路由/推荐的高年级判定，容错足够）；`MajorDestinationStats` 预留 `@Scheduled` 预计算缓存位（本期数据量小直接查询）。

**假设与简化**：① `listVerifiedUsersByMajor` 以 `auth_status=VERIFIED` 为候选门槛（M1 未暴露 `isActive` 契约，`DISABLED` 极端情况留待 UserService 扩展）；② 路径卡 `version` 修订留痕以乐观锁号最小实现，不单建历史表（对齐 §3.5）；③ 推荐"匹配校友"以登录名为展示句柄，专属昵称字段为未来工作；④ 文件上传仅落 URL 字符串，复用全局上传服务。
