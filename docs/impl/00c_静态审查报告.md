# 代码-设计一致性静态审查报告

> 审查对象：`backend/src/main/java/com/xju/sem/**` 实现 + `schema.sql` ↔ `docs/design/01–07` 详细设计 + `frontend/src/**`
> 输入：8 名审查员上报的 119 条"代码 vs 设计"不一致（JSON），并纳入 `00_编码一致性报告.md`、`00b_扩展编码一致性报告.md` 已登记的 6 项待对接债、2 项治理注记与 1 项已知必修（`OpportunityServiceImpl.update` 内推审核绕过）。
> 处理：去重合并同类 → 按 severity 排序 → 归类六 area → 逐条给出处置（改代码 / 改设计文档 / 改前端 / 记为已知偏差）。
> 合并结果：**119 条原始 → 104 条**（15 条为同源连带项被并入根因条目）。

---

## 一、总体结论

**一句话结论：后端"代码 ↔ schema.sql"高度自洽，绝大多数严重/中不一致是"设计文档滞后于 v3.1 reconcile 后的 schema"的文档债（处置以回填文档为主）；真正需要改代码的仅约 13 项风控/校验/状态机缺口；前端 ↔ 后端 才是当前唯一的联调阻断层，12 条严重路径/字段错配会让核心流恒 401/400/404。**

分层评价：

1. **后端内部（代码 ↔ schema.sql）：吻合度高。** `00/00b` 已用静态交叉核对证实：24 个跨模块契约方法两端签名/参数/返回类型逐一对齐、4 条领域事件（采纳 / 认证提交 / 知识提交 / 机会提交）字段闭合、BaseEntity/Result/低耦合地基约束全绿、0 编译级签名错误、0 禁用论坛词。本轮上报的大量"严重/中"数据表字段与命名不一致，几乎都是 **`schema.sql` 头部已声明 reconciled、但 `docs/design/01–07` 正文未同步** 的单侧文档滞后，代码与 schema 二者是自洽的。**这类处置一律是"改设计文档回填"，不是缺陷。**

2. **需要改代码的是一小撮硬伤（共约 13 项）**，集中在风控与业务正确性，务必在验收前处理：双人担保被弱化为单人确认（S3）、alumni_path_card 缺唯一键 + 无重复校验（S6）、深造录取方式 admission_type 完全缺失（S7）、M5 审核态并入主状态机破坏正交裁决（S17）/ 无 REJECTED 态致被拒机会不可重提（S18）/ team_required 校验缺失（S19）、M7 治理契约四方法未留桩（S15）、studentNo 无格式校验（C5）、gpa_scale 硬编码（C11）、备考资料字段漏实现（C10）、冷启动未按同学院收窄（C16）、补救排序 tie-break 方向写反（C38），以及 `00b` 已登记的 `OpportunityServiceImpl.update` 内推审核绕过（C51，中危）。

3. **前端 ↔ 后端：吻合度低，是真正的阻断项。** 12 条严重全部为"前端按理想/臆测的 API 契约编写、从未与真实后端对表"：登录 token 字段名错（全站 401）、注册/求助/采纳/审核/进度标记路径或请求体错（恒 400/404）、详情页把包裹 DTO 当实体直接用（字段全空）。**这一层需前端按后端实际契约整体返工**，且需 M7 补一个专用审核吞吐统计端点（FS12）、后端补 2 个展示冗余字段（C45/C50 建议改后端）。

4. **架构方向正确、债务已被登记。** "统一审核队列（`/api/v1/audit-tasks/{id}/decide`）取代各模块独立 approve/return/force-end 路由"、"timeline 三表归 M6 owner"等，都是 `08` 集成报告已裁决的既定架构；设计文档 01/03/05/07 正文未同步，属文档债而非实现错误。6 项"契约已就绪、调用方仍走临时本地实现"的待对接迁移债均有源码 TODO 佐证，见第四节。

**吻合度评分（主观）：后端代码↔schema 90/100；后端代码↔设计文档 55/100（文档滞后为主）；前端↔后端 35/100（需返工）。**
