## 1. 模块功能说明（做什么、核心流程，白话）

M5 管理"机会信息聚合"与"组队撮合"两条主线，围绕 `opportunity / team / team_member` 三张表展开：

1. **机会发布与类型化展示**：已认证的校友/管理员发布竞赛/大创/实习/讲座类机会（`type`）。`is_referral=1`
   （内推类）的机会需 M7 终审通过才对外公开，其余机会按 `deadline` 即时计算状态直接对外可见——
   审核门与对外时间窗口**合并在同一个 `status` 字段**表达（详见 §4.1 与"假设与简化"）。
2. **机会状态机**：`PENDING_REVIEW → ONGOING → CLOSING_SOON → CLOSED → ENDED`，由 `@Scheduled`
   定时任务按 `deadline` 自动推进；发布人/管理员可随时手动结束，管理员可不受状态限制强制下线（治理）。
3. **机会终审**：`OpportunityService.approve/reject` 供 M7 统一审核队列调用（策略路由，与 M1/M3 终审接口
   同一模式），终审通过/拒绝后站内通知发布人；`is_referral=1` 的机会创建时发布
   `OpportunitySubmittedEvent`，供 M7 侧建 `audit_task(target_type=OPPORTUNITY)`。
4. **报名信令**：不组队/不内推的机会以"我要报名"按钮做热度信令（本期简化为校验+通知，见 §4）。
5. **组队 CRUD 与队伍状态机**：`RECRUITING → FULL → ONGOING → ENDED`。队伍可挂靠某条机会
   （校验机会处于 `ONGOING/CLOSING_SOON`），也可不关联任何机会自由组队（schema `team.opportunity_id`
   注释"可空,自由组队"）。
6. **申请加入 / 审批 / 退出 / 移除**：`team_member` 记录申请与审批关系，`team.current_size` 的并发名额
   竞争走 Mapper 级 CAS（`WHERE current_size < capacity`）防超员，审批通过时在单个 `@Transactional`
   方法内原子完成"计数 CAS + 状态 CAS + 满员连带转 FULL"。
7. **机会 ENDED 级联结束队伍**：机会进入 `ENDED`（无论手动结束还是定时归档）时，其下非终态队伍统一转
   `ENDED` 并通知全体已加入成员。
8. **列表与组队广场**：机会列表支持类型筛选/关键字/"即将截止"专区/"显示已截止已结束"开关；组队广场
   默认仅展示 `RECRUITING`，交互为结构化筛选列表，不做论坛式信息流（沿用"组队广场"页面命名，非架构概念）。
9. **对外契约**：`OpportunityService.getBrief/approve/reject` 供 M6 时间线节点只读引用、首页仪表盘
   "即将截止机会"卡片、M7 终审队列调用。

**边界（不做的事）**：不做内推申请单业务（`referral_ticket` 表已在 schema 中保留，本期不写 Service/Controller）；
不做真实报名全流程与材料收集（外链跳转由前端负责）；不做队伍内部协作工具（任务看板/群聊）；不做机会/队伍
标签体系与结构化匹配推荐；不做运营数据统计报表呈现。

## 4. 关键实现点

### 4.1 审核门与状态机合并（§6.1，相对 05 文档最大的裁剪）
05 文档设计为"`status`（四态对外时间窗口）+ `audit_status`（正交三态审核门）"双字段模型，但
`schema.sql` 的 `opportunity` 表**只有一个 `status` 列**，注释直接给出五态
`PENDING_REVIEW/ONGOING/CLOSING_SOON/CLOSED/ENDED`（与本任务书要求的状态机完全一致）。因此本实现
把"审核门"合并进同一个字段：`is_referral=1`（内推类）发布时直接落 `PENDING_REVIEW`，其余机会创建时
按 `computeStatusByTime(deadline)` 直接计算落点对外可见，不经审核。终审 `approve` 时同样调用
`computeStatusByTime` 而非硬编码 `ONGOING`，避免"审核排队很久、deadline 都快到了却还判给 ONGOING"
的错误。`reject` 因 schema 无独立 `REJECTED` 态，直接终态归档为 `ENDED`（理由随通知下发+日志留痕，
不入库），与 `help_ticket` 的 `close_reason` 处理方式同一原则。

### 4.2 状态类流转的并发控制（CAS，无 version 列）
`opportunity/team/team_member` 均无 `version` 列，所有状态流转统一走"带 `WHERE 状态=?` 条件的 CAS
UPDATE"（`OpportunityMapper.casStatus`/`TeamMapper.casStatus`/`TeamMemberMapper.casJoinStatus`），
受影响行数为 0 即判定"前置状态不满足或并发被他人先改"，转 `STATE_CONFLICT`。内容编辑（`update`）走
"归属校验 + 普通 `updateById`"，与状态流转分工，和 M1/M3/M4 先例一致。

### 4.3 机会状态定时任务（§6.2）
`OpportunityStatusScheduler` 每 10 分钟批量执行两级推进（`ONGOING→CLOSING_SOON`、
`(ONGOING,CLOSING_SOON)→CLOSED`，纯批量 CAS UPDATE，不逐行处理），随后扫描 `CLOSED` 且已超归档窗口
（`deadline + archive-days`，默认 14 天，可配置）的候选集，逐行 `archiveIfClosed` CAS 后触发
`TeamService.endAllByOpportunity` 级联与通知——按行 CAS 而非批量，是为了准确知道"这一行确实是本次
处理的"，才去联动下游（级联结束队伍、发通知），避免与并发的手动结束重复处理，与
`KnowledgeEntryExpiryScheduler`/`HelpTicketAutoCloseJob` 同一纪律（批量各自提交、不锁全表）。

### 4.4 队伍名额并发（§6.5，单事务双重 CAS）
`TeamMemberServiceImpl.approve` 在**同一个 `@Transactional` 方法**内依次：①`team.current_size`
CAS 递增（`WHERE current_size < capacity`，防超员，0 行即 `LIMIT_EXCEEDED`）；②`team_member.join_status`
CAS `APPLYING→JOINED`（0 行即 `STATE_CONFLICT`，触发整体回滚，含①已做的递增）；③重新查询后若
`current_size>=capacity` 则 `team.status` CAS `RECRUITING→FULL`。三步任一失败均靠事务回滚而非手工
补偿，与 05 文档"approve 单事务原子完成"的要求一致。

### 4.5 机会 ENDED 级联结束队伍（§6.7）
`TeamService.endAllByOpportunity(opportunityId, reason)` 被两处调用：①`OpportunityServiceImpl.end`
（手动结束/强制下线，无论谁触发都级联）；②`OpportunityStatusScheduler`（定时归档触发）。查出该机会下
非终态队伍逐条 CAS 转 `ENDED`，仅对实际 CAS 成功的行发送"所属机会已结束"通知，避免与队长手动同时结束
产生重复通知。

### 4.6 组队的机会关联可空（自由组队）
`team.opportunity_id` 在 schema 中可空（注释"关联机会(可空,自由组队)"），与 05 文档"仅
`team_required=1` 的机会下可创建队伍"的强绑定设计不同——本 schema 未开 `team_required` 列。因此
`TeamController` 同时暴露 `POST /opportunities/{opportunityId}/teams`（挂靠机会，校验其处于
`ONGOING/CLOSING_SOON`）与 `POST /teams`（自由组队，`opportunityId=null`，不做任何机会状态校验）
两个入口，`TeamServiceImpl.createTeam` 共用同一实现，`opportunityId` 为 null 时跳过机会校验分支。

### 4.7 假设与简化（相对 05 文档，以 schema.sql 为准）
1. **opportunity 无 `audit_status/organizer_name/applicable_scope/location/external_url/team_required/
   start_time/event_end_time/archive_at/end_reason/review_comment/apply_count/view_count` 等列**：
   审核门合并进 `status`（4.1）；`archive_at` 改为运行时按 `deadline+archive-days` 计算，不落库；
   `end_reason`/终审 `review_comment` 仅随通知下发+日志留痕，不入库（与 `help_ticket.close_reason`
   同一处理方式）；`apply_count` 缺失导致"简单报名信令"简化为**只做状态校验 + 通知发布人，不做热度
   计数持久化**——如需真实计数看板，需在 `opportunity` 追加 `apply_count INT DEFAULT 0` 列（纯新增列，
   向后兼容，不影响现有数据/代码）。
2. **team 无 `full_reason/end_reason/started_at/ended_at` 列**：`FULL→RECRUITING` 的隐式回退本期
   **统一处理**（不区分自动满员 AUTO 与队长手动锁定 MANUAL_LOCK，退出/移除后一律尝试回退），是相对
   05 文档 §6.6"仅 AUTO 才回退"的简化；`started_at/ended_at` 改用 `BaseEntity` 的 `updatedAt`
   近似定位关键时间点。
3. **team_member 无 `apply_message/reviewed_at/reviewed_by` 列**：不存申请自我介绍与审批人快照，
   `createdAt/updatedAt` 兼任"申请时间/最近一次状态变更时间"语义；`REJECTED`/`REMOVED` 合并为单一
   `LEFT` 终态（`quit`/`remove` 两个业务动作落到同一个 DB 状态值，仅操作人校验不同），无法从数据上
   区分"主动退出"与"被移除"，相对 05 文档 §3.3 五态设计的裁剪。
4. **内推 `referral_ticket` 本期不实现业务**：表已在 schema 中保留，本模块不写对应
   entity/mapper/service/controller，与 08/09 设计文档"内推功能本期不做，标 Could"的裁决一致。
5. **机会终审 Controller 不在本模块暴露**：`approve/reject` 由 M7 治理端统一审核队列
   （`/api/v1/audit-tasks/{id}/decide`，经 `AuditTargetHandler` 策略路由）调用本模块 `OpportunityService`，
   与 `KnowledgeEntryController`/`AuthApplicationController` 的终审接口同一模式；M7 侧
   `OpportunityAuditHandler`（注册到 `AuditTargetType.OPPORTUNITY`）与 `AuditTaskEventListener` 监听
   `OpportunitySubmittedEvent` 两处接线留给"M7剩余"任务补充，本模块已按冻结契约签名就绪
   （`void approve(Long id, Long reviewerId)` / `void reject(Long id, Long reviewerId, String reason)`）。
6. **未做的对外方法**：内推相关 `ReferralTicketService` 全套接口本期未定义（无调用方，见第4点）；
   机会/队伍标签化匹配推荐、运营统计报表均未实现，留待后续迭代按需补齐。
7. **依赖前提**：`UserService`（M1）、`NotificationService`（通知模块）均已被现有代码引用，属既有跨
   模块契约；所有对外通知调用都包了 try/catch，发送失败不阻断主流程。
