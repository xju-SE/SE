## 3. 地基约束（BaseEntity / Result / 低耦合）

**3.1 entity 继承 BaseEntity / 无重复声明审计字段**

- `BaseEntity`（`common/BaseEntity.java`）统一持有 `id`(@TableId AUTO) / `deleted`(@TableLogic, select=false) / `createdAt` / `updatedAt`。
- 核心闭环 entity 均 `extends BaseEntity` 且 `@EqualsAndHashCode(callSuper=true)`，**未重复声明** id/deleted/createdAt/updatedAt：`User`、`HelpTicket`、`HelpAnswer`、`KnowledgeEntry`、`AuditTask`、`Notification`、`KnowledgeFeedback`、`AuthApplication` 等。
- **3 个 entity 有意不继承 BaseEntity**，且经核对为**正确设计**（对应表缺少完整审计列，若继承会向不存在的列写值）：
  - `HelpFollowup`（`help_followup` 表无 `updated_at`）：自持 id/deleted/createdAt(`FieldFill.INSERT`)，与 schema 列精确匹配，类注释已说明。
  - `HelpRoute`（`help_route` 表仅 id/ticket_id/matched_user_id/match_score/status/notified_at，无 deleted/created_at/updated_at）：自持 id + 业务列，无 @TableLogic（表本就无软删），与 schema 精确匹配。
  - `MockStudentRoster`（模拟学籍参照表）。
  - → 这类"精简表不套 BaseEntity"是刻意规避"向不存在列写值"的正确处理，**非缺陷**。

**3.2 Controller 统一返回 Result**

已核对全部 12 个 Controller 的每个 mapping 方法返回类型，**100% 为 `Result<...>`**（含 `Result<Void>`）：`AuditTaskController`、`Help*Controller`、`Knowledge*Controller`、`NotificationController`、`Auth*Controller`、`InviteCodeController`、`UserController`。无一处裸返回实体/DTO。

**3.3 无跨模块直接访问别模块 Mapper/表（低耦合）**

- 逐模块 grep：`module/{user,knowledge,help,admin,notification}` 中**没有任何文件 import 其它模块的 `.mapper.` 类**。
- 跨模块也**未 import 别模块的 entity**（`help/admin/knowledge/notification` 均不 import `user.entity`；`admin/knowledge` 不 import `help.entity`；等）。
- M7(admin) 对 M1/M3 的依赖**一律经 Service 接口**：`AuditTaskServiceImpl`/`PreCheckServiceImpl` 只注入 `KnowledgeEntryService`/`AuthApplicationService`/`UserService`，调用 `getById/getBrief/approve/returnToCandidate`（见 §6-A 治理注记）。跨模块耦合仅停留在 Service 接口 + 事件 + DTO 层，符合地基低耦合要求。

## 5. 禁用论坛词 / 编译级明显错误

- **禁用论坛词**：grep `论坛/帖子/发帖/回帖/楼主/楼层/板块/版块/点赞/盖楼/贴吧` over `module/**` → **0 命中**。术语统一为"求助单 / 回答 / 采纳 / 追问 / 知识条目 / 路由"，符合"结构化求助（非论坛）"定位。
- **通知 type 取值合法性**：所有 `send(...)` 的 type 实参均落在 `NotificationType` 值域 `HELP_MATCH/ADOPT/AUDIT_RESULT/SYSTEM` 内——常量 `HelpConstants.NOTIFY_HELP_MATCH/NOTIFY_ADOPT/NOTIFY_SYSTEM`（值 = "HELP_MATCH"/"ADOPT"/"SYSTEM"）与字面量 "AUDIT_RESULT"/"SYSTEM" 均合法；`send` 内 `NotificationType.isValid` 兜底校验。
- **编译级检查**（静态）：契约方法/事件构造器/DTO getter 三处逐一核对无缺 import、无类名不符、无参数个数/类型对不上；接口与实现 `@Override` 签名逐字一致。**未发现编译级签名类错误。**
- 说明：本次为静态交叉核对，未执行 `mvn compile`（`backend/pom.xml` 存在，全量编译需拉取依赖，超出本轮一致性审查范围）。

## 6. 补充观察（低危，非阻断）

**注 A（契约边界治理，低危）**：M7(admin) 依赖了两个**未被声明为"冻结契约"的 `getById`**——
- `knowledgeEntryService.getById(Long,Long,boolean)`：`PreCheckServiceImpl:43`、`AuditTaskServiceImpl:185`；而 `KnowledgeEntryService` 接口 javadoc 明确写"getById 等为本模块内部方法，签名可自由演进，不受跨模块约束"。
- `authApplicationService.getById(Long)`：`AuditTaskEventListener:67`、`AuditTaskServiceImpl:183/403`；同样未列入 M1 §8 冻结契约。

当前**签名对齐、可编译、运行无碍**（审核详情页确需完整 detail，`getBrief` 的轻摘要不够用）。风险在"未来 M3/M1 若演进 getById 签名会静默打断 M7"。建议把这两个 `getById` 提升为文档化契约方法（或新增专用 detail 契约），消除"契约冻结声明"与"实际跨模块依赖"的口径不一致。

**注 B（运行期验证项，低危）**：采纳链存在**嵌套 AFTER_COMMIT 发事件**——`adopt`(tx) → `HelpAnswerAdoptedListener`(AFTER_COMMIT，无 @Transactional) → `createFromHelpAdoption`(新 REQUIRED 事务) → `doSubmit` 内发 `KnowledgeEntrySubmittedEvent` → `AuditTaskEventListener`(AFTER_COMMIT) 建 audit_task。因 `createFromHelpAdoption` 确会开启**全新事务**，其提交理论上能正常触发下一个 AFTER_COMMIT 监听，代码注释也已言明该设计；但"AFTER_COMMIT 回调里再发 TransactionalEventListener 事件"是 Spring 易踩点，建议加一条集成测试确认"求助采纳生成的知识候选确实生成了对应 audit_task"，坐实链1→链3→M7 全链贯通。

**注 C**：`KnowledgeEntryService.existsPublished` 与知识侧 `getBrief` 属 M6 预留契约，`existsPublished` 现无活调用点（M6 未落地），签名已就绪，**符合预期非缺陷**。
